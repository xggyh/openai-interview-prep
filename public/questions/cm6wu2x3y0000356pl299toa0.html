Design YouTube

System Design Senior 原题 ↗
OpenAI · Senior · 📋 1 · 🕒 Late October, 2025

Problem Statement

Design a video streaming platform like YouTube where creators can upload and share videos, viewers can stream content on-demand, and users receive personalized recommendations based on their viewing history and preferences.

我的思路与解法

题目本质

设计 YouTube —— 视频上传 + 多分辨率 transcoding + 全球流式播放 + 个性化推荐。OpenAI 报告 Senior 级。

经典超大型 SD 题。考点：视频处理 pipeline、CDN、推荐系统、metadata 存储。

需求拆解

功能性：

  • Creator：上传视频、设置标题/描述/缩略图、查看分析
  • Viewer：搜索 / 浏览 / 播放视频、订阅、评论、点赞
  • 推荐：首页 / 相关视频
  • 多分辨率自适应（240p → 4K）

非功能性：

  • 5 亿 DAU，30 亿播放/天
  • 上传：500 hours/min
  • 播放延迟 < 2s（首帧）
  • 99.99% 可用

容量估算：

  • 500 hours/min × 60 = 30,000 min/min 上传 → 1 hour ≈ 1 GB raw → ~30 TB/min 上传
  • 流量：30 亿播放 × 平均 10 min × 5 Mbps = 9 PB/day egress（主要走 CDN）
  • 存储 5 年 ≈ 几 EB（用冷存储分层）

整体架构

Creator │ upload ▼ ┌──────────────┐ │ Upload API │ → presigned S3 multipart └──────┬───────┘ │ raw video in S3 ▼ ┌──────────────┐ │ Transcoding │ → ffmpeg pipeline │ Worker Pool │ 生成 240/480/720/1080/4K + HLS chunks └──────┬───────┘ │ ▼ ┌──────────────┐ │ S3 (Frames + │ chunks 4-6 s 一段 + manifest │ Manifests) │ └──────┬───────┘ │ ▼ ┌──────────────┐ │ CDN │ Cloudflare / 自建 PoP │ (edge cache) │ └──────────────┘ Viewer │ ▼ ┌──────────────┐ │ Edge │ → returns nearest CDN URL └──────┬───────┘ │ ▼ pull HLS / DASH video player (ABR adaptive bitrate) Side: Metadata, Search, Recommendation ┌──────────────────────┐ │ Metadata API │ → Postgres + Cassandra │ Search (Elastic) │ │ Recsys (offline + │ │ online ranking) │ └──────────────────────┘

核心组件

1. 上传流程

Creator → POST /upload/init → 收到 upload_id + S3 presigned multipart URLs
Creator → PUT each 5MB chunk directly to S3
Creator → POST /upload/complete → notify backend
Backend → enqueue transcoding job → workers pick up
Workers → produce HLS variants → S3
Workers → notify metadata service → publish video

直传 S3，绝不走 backend 服务转发流量（成本爆炸）。

2. Transcoding Pipeline

每个 video 转成多套：

| Resolution | Bitrate | Codec | |---|---|---| | 240p | 400 kbps | H.264 / H.265 | | 480p | 1 Mbps | H.264 / H.265 | | 720p | 2.5 Mbps | H.264 / H.265 | | 1080p | 5 Mbps | H.264 / H.265 / AV1 | | 4K | 25 Mbps | H.265 / AV1 |

每个 variant 切成 4-6 秒的 chunks → 生成 HLS playlist (.m3u8)。

并行化：原视频按时长切 N 段 → fan-out 给 N worker 各自 transcode → fan-in 合并 manifest。

GPU 加速：NVIDIA NVENC / Intel QSV，比 CPU x264 快 10x。

3. CDN & Adaptive Bitrate

播放器请求 HLS manifest → 包含 variants 列表 → 播放器根据当前带宽选 bitrate：

#EXTM3U
#EXT-X-STREAM-INF:BANDWIDTH=1280000,RESOLUTION=640x360
360p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=2560000,RESOLUTION=1280x720
720p/playlist.m3u8

每个 chunk 独立可缓存 → CDN edge 缓存命中率高（top 1% 视频占大半流量）。

4. 存储分层

| 层 | 媒介 | 数据 | |---|---|---| | Hot | SSD (CDN edge) | 上传 7 天内 + 高 popularity | | Warm | S3 Standard | 7-90 天 + 中等 popularity | | Cold | S3 Glacier | 90+ 天且 < 100 view/month | | Archive | Glacier Deep | 1 年+ 几乎无人看 |

冷数据请求时主动重新生成 hot copy：第一个查的人等 30 秒回热，后续秒开。

5. Metadata Schema

-- Postgres (videos / users / channels)
CREATE TABLE videos (
  id              UUID PRIMARY KEY,
  channel_id      UUID,
  title           TEXT,
  description     TEXT,
  duration_sec    INT,
  manifest_url    TEXT,
  upload_status   TEXT,
  published_at    TIMESTAMPTZ,
  view_count      BIGINT,           -- eventual
  like_count      BIGINT
);

-- Cassandra (views / engagement events, write heavy)
CREATE TABLE views (
  video_id   UUID,
  ts         TIMESTAMP,
  user_id    UUID,
  watch_pct  FLOAT,
  PRIMARY KEY ((video_id), ts)
);

6. 推荐系统

两阶段：

  1. 召回（candidate generation）：从亿级 video 中选 hundreds
  • Collaborative filtering（看过类似 video 的用户也看了 X）
  • Content-based（embedding 相似）
  • Subscribed channels' new uploads
  1. 排序（ranking）：DNN 模型对 hundreds 排序，输入特征：
  • User 历史（last 50 watches）
  • Video freshness、CTR、watch time
  • Real-time signals（last 5 min behavior）

离线训练（每天 / 每小时），在线推理 (TF Serving / Triton)。

7. Search

  • 视频 metadata + transcript（通过 ASR 自动转）→ Elasticsearch
  • 搜索结果 ranking：BM25 + 用户行为 features

8. Engagement / Analytics

每 view / like / comment → Kafka → 实时聚合到 Cassandra (counter) + 离线 ETL 到 ClickHouse。

取舍

| 决策 | 选择 | 替代 | |---|---|---| | 视频协议 | HLS / DASH (HTTP-based) | RTMP：UDP，不友好穿透 | | 上传 | 客户端 multipart 直传 S3 | 走 backend：流量打爆 | | Transcoding | 离线 fan-out | 实时：cost 爆 | | Hot/Cold | 4 层分层 | 单层全 SSD：贵 | | View count | Cassandra counter eventual | DB row INCR：写热点 |

关键技术细节

  • AV1 vs H.264/H.265：AV1 比 H.264 省 ~30% 带宽，但编码 10x 慢。Top videos 用 AV1，长尾不值得
  • Watermark / DRM：高价值内容 + Widevine / FairPlay DRM
  • Live streaming：单独 pipeline，sub-second 延迟用 WebRTC / RTMP + low-latency HLS
  • Geo-replication：每 video 复制到 multi-region S3，CDN edge 拉最近
核心要点

三大要点：(1) 客户端直传 S3 + ffmpeg 离线 transcoding pipeline；(2) HLS + CDN edge 缓存 + ABR 自适应；(3) 推荐两阶段（召回 + 排序）。OpenAI 面试官可能追问编码 codec 取舍和 GPU 加速。

易错点 / Red Flag

❌ 上传走 backend —— 流量打死 server； ❌ 不切 chunks 整个视频一个文件 —— CDN 缓存粒度太大、无法 ABR； ❌ View count 用 RDBMS row UPDATE —— 写热点； ❌ 用户全部 history 跑推荐 —— 模型太慢；用 recency + recall stage 缩减； ❌ 单一 codec —— 老设备不兼容或新设备没用上 efficient codec。

延伸 / Follow-up

"如何 Live streaming？" → 推流 RTMP → ingest server → 实时 transcode (HLS sub-1s latency) → CDN；"如何降低 view 延迟？" → CDN preloading top videos at edge；"如何检测违规内容？" → ASR + frame sampling → ML classifier，超阈值 enqueue 人工审核。
