Design a News Aggregator System

📚 教学版深度讲解 这道题已重写为新手友好的完整教学版：含概念铺垫、需求拆解、容量估算、架构 step-by-step 推演、组件深挖、45 分钟面试节奏、样板讲解稿、Follow-up Q&A、易错点 + 加分项。约 600 行。
System Design Staff 原题 ↗
Google · Staff · 📋 1 · 🕒 Late November, 2025

Problem Statement

Design a news aggregation system like Google News that allows users to subscribe to different news sources, categories, and topics, then view a personalized real-time news feed based on their interests.

我的思路与解法

1. 题目本质

News Aggregator = 抓取全球新闻源 → 去重 → 聚类同事件 → 个性化推荐 → 实时分发。

典型产品：

  • Google News —— 1.5B users，跨语言/区域
  • Apple News —— iOS 集成
  • Flipboard —— magazine UX
  • SmartNews —— 日本起家
  • TopBuzz / 今日头条 —— 推荐算法标杆
  • Yahoo News

为什么这是 STAFF 题（1 ppl 报告但概念硬核）：

考的是 4 个独立子系统的协调：

  1. Crawling at scale —— 10k+ news sources, 多语言
  2. Deduplication + clustering —— 不要 5 篇同一新闻
  3. Personalization —— 1B users 个性化首页
  4. Real-time delivery —— 突发新闻 5 分钟内推达

考 STAFF 关键：不是 simple feed system，而是"四个子系统 + ML pipeline + multi-region 分发"的全栈。

3. 容量估算

  • Sources: 10k news sites + 100k blogs + RSS = 1M crawl targets
  • Articles/day: 1M new (after dedup → 100k unique events)
  • Storage: 1M × 5KB = 5GB/day → 2TB/year text + 10TB/year images
  • Users: 1B MAU × 50% DAU = 500M DAU
  • Feed reads: 500M × 10 reads/day = 5B/day = 60k QPS sustained, 200k peak
  • Crawl rate: 1M sources × 1 visit/hour = 280/sec sustained

4. 高层架构

Step 1: Crawler

URL Frontier (Kafka) → Worker Pool → Page Fetcher → HTML Parser
                                                          ↓
                                              Raw Article Store (S3)
                                                          ↓
                                                  Article Queue (Kafka)

Polite crawling:

  • robots.txt 解析
  • Per-domain rate limit (1 req/s per source)
  • User-Agent 标识
  • 304 Not Modified caching

Distributed scheduling: URL frontier 用 priority queue (新闻源每 5 分钟 revisit, 长尾博客每天一次)。

Step 2: Content Extraction

HTML → clean article text + metadata：

  • Boilerplate removal：删 nav / ads (use libraries like Readability / Goose3)
  • Metadata: title, author, publish_time, language
  • Image extraction: og:image / first inline image
  • Language detection: fastText

Step 3: Dedup + Clustering

Layer 1: Exact dedup (URL canonicalization)

  • Normalize URL (remove utm params, sort query string)
  • Hash → check Bloom filter

Layer 2: Near-dup (SimHash / MinHash)

  • Compute SimHash of article body
  • Hamming distance < 3 → considered dup

Layer 3: Semantic clustering (Event grouping)

  • Embedding via sentence-BERT
  • HDBSCAN / agglomerative cluster
  • "Same event" = articles in same cluster within 24h window
"Apple Vision Pro 发布" cluster:
  - Apple official article
  - TechCrunch article
  - Verge article
  - 5 ja translations
  → 1 cluster, show as 1 entry with "20 sources"

Step 4: Storage Schema

article (raw):
  article_id, url, title, body, lang, publish_time, source_id, cluster_id?

cluster:
  cluster_id, topic, lang, primary_article_id, num_sources, score, created_at

user_feed (per user, sharded):
  user_id, article_or_cluster_id, score, position, served_at

Step 5: Personalization Pipeline

User profile (interests, history) ↘
                                    Recommendation Service
Article catalog (with embeddings) ↗         ↓
                                    Ranked feed (cached)

Two-tower model:

  • User tower: embed user from history (read articles, dwell time, likes)
  • Item tower: embed article from text + metadata
  • Score = cosine similarity

Retrieval: top-1000 candidates via ANN (FAISS) → ranking model (DNN with features) → top-N feed。

Step 6: Real-time Delivery

Breaking news flow:

Crawler finds breaking story
  → Dedup check (within last 1 hour)
  → If new + high "newsworthiness" score:
      → fan-out to user feeds (push notifications + insert at top)
      → CDN cache invalidate

Newsworthiness signal:

  • Velocity: how many sources publish in 5 min
  • Editor curation (manually flag)
  • ML model (text + meta) classifies "breaking"

5. 组件深挖

Deep Dive 1: Dedup Pipeline (SimHash)

def simhash(text):
    tokens = tokenize(text)
    weights = tfidf(tokens)
    vec = [0] * 64
    for tok, w in zip(tokens, weights):
        h = hash64(tok)
        for i in range(64):
            vec[i] += w if (h >> i) & 1 else -w
    return sum((vec[i] > 0) << i for i in range(64))

def is_dup(h1, h2, threshold=3):
    return popcount(h1 ^ h2) <= threshold

Storage: 1M articles × 8B SimHash = 8 MB → 全 in memory，scan O(N) per query 也快。

Optimization: split 64-bit hash into 4 chunks of 16-bit each → index by chunk → only compare articles sharing ≥1 chunk → 10× speedup.

Deep Dive 2: Clustering — Event Grouping

Online clustering (新文章来了实时归簇):

def assign_cluster(article, recent_clusters):
    emb = embed(article)
    best_cluster, best_sim = None, 0
    for c in recent_clusters:  # within last 24h
        sim = cosine(emb, c.centroid)
        if sim > 0.85 and sim > best_sim:
            best_cluster, best_sim = c, sim
    if best_cluster:
        best_cluster.add(article)
        best_cluster.centroid = update(best_cluster)  # incremental mean
    else:
        new_cluster(article)

Recent clusters store: Redis hash, keyed by language + topic prefix。

Batch reclustering: 每天跑一次 HDBSCAN 整理（online algo 漂移）。

Deep Dive 3: Personalization Two-Tower

                ┌─────────────────┐
User behaviors → │ User Tower (DNN)│ → user_vec [128]
                └─────────────────┘
                                          ↘
                                            cosine sim → score
                                          ↗
                ┌─────────────────┐
Article text → │ Item Tower (BERT)│ → item_vec [128]
                └─────────────────┘

Training: pairs of (user, positive article, negative article) with sampled negatives, hinge loss.

Inference:

  • User vec computed offline (every 1h)
  • Item vec computed at ingest (once per article)
  • Retrieve top-1000 via FAISS ANN
  • Re-rank top-100 with deeper DNN (more features)

Deep Dive 4: Real-time Index

Articles 每分钟新增 ~700。Index 选择：

  • Elasticsearch for keyword + facet search
  • FAISS / Milvus for embedding ANN (article → similar articles, user → matching articles)
  • Redis for hot store of last 24h articles by cluster

Refresh latency: write to ES + FAISS in same transaction (eventual consistent), refresh < 1 min。

Deep Dive 5: Multi-language

100 languages，怎么处理？

Storage: 每篇文章带 lang field

Personalization:

  • User profile 包含 language preferences
  • Filter retrieval by lang at FAISS query time

Translation:

  • 对 trending events，translate top article to user's lang on-demand (cache)
  • 小语种 fallback to English version

Cross-lingual clustering: same event 多语言 cluster - 用 multilingual embedding (LaBSE)。

Deep Dive 6: Content Moderation

Filter:

  • Spam / clickbait classifier (text + meta)
  • Misinformation flag (cross-reference with fact-check DB)
  • NSFW image filter
  • Geo-restricted content (some country bans certain topics)

Trust score per source: 长期表现好的源加分，违规多的降权 / 移除。

Deep Dive 7: Caching Strategy

3 tiers:

  1. CDN edge: top trending in region (cache 30s)
  2. Redis hot store: per-cluster current best article (cache 5 min)
  3. Personalized feed cache: per-user feed (precompute every hour, cache 1h)

Cache invalidate on breaking news: targeted invalidate by region + category。

时间阶段
0-5min澄清：sources? languages? real-time?

7. 样板讲解稿

News aggregator 实际是 4 个独立 pipeline 串起来：(1) crawl, (2) extract+dedup+cluster, (3) personalize, (4) deliver.

架构：

  1. Crawler: URL frontier 队列，10k news + 100k blogs，per-domain rate limit
  2. Extraction: HTML → clean text + metadata (Readability lib)
  3. Dedup: 3 层 — exact URL → SimHash near-dup → semantic clustering via embedding + HDBSCAN
  4. Personalization: two-tower DNN (user + item embed)，FAISS retrieve top-1000 → DNN rerank top-100
  5. Real-time: breaking news pipeline (newsworthiness ML + fan-out)

Storage:

  • Raw articles: S3 (cheap)
  • Hot last-24h: Redis (per-cluster)
  • Search: Elasticsearch
  • ANN: FAISS / Milvus
  • User feed: precomputed, cached per user

Multi-language: lang filter at retrieval + cross-lingual clustering (LaBSE embedding)

Numbers: 1B users, 1M articles/day → 100k unique events, 60k QPS sustained feed read, 200k peak.

8. Follow-up Q&A

Q1: "Crawler 怎么不 DDOS 新闻站点？"

A：3 层 politeness：

  1. Honor robots.txt (Crawl-delay directive)
  2. Per-domain rate limit (e.g., 1 req/s, 5s back-off on 429/503)
  3. Smart scheduling: 新闻站点每 5 min 一次，长尾博客每天

Q2: "同一新闻 5 篇文章 怎么去重？"

A：3 层：

  1. URL canonical → Bloom filter exact dedup
  2. SimHash hamming distance < 3 → near-dup
  3. Embedding cluster (cosine > 0.85, within 24h) → "same event"

最终 user 看到的是 cluster，不是单篇文章。

Q3: "Breaking news 5 min 内到 user feed 怎么做到？"

A：

  • Crawler 每 30s scan high-priority sources
  • ML classifier 5s 内判 newsworthiness
  • 通过 Kafka → push 服务 fan-out（仅订阅相关 category 的 user）
  • CDN invalidate breaking story 区域 cache

End-to-end 通常 2-3 min。

Q4: "1B users，怎么 precompute personalized feed？"

A：

  • 不是真 precompute 1B 个完整 feed（不可能存）
  • Precompute per user: user_vec (128 dim) — 每小时 batch
  • Inference time: FAISS retrieve top-1000 + DNN rerank top-100 → < 100 ms
  • Cache feed in Redis 1h，passive refresh on user activity

Q5: "Personalization model 怎么训练？"

A：

  • Negative sampling from impressions（show but not click）
  • Positive: click + dwell > 30s
  • Hinge loss / triplet loss on (user, pos, neg)
  • Daily retrain on yesterday's data
  • A/B test 新模型 5% traffic

Q6: "如果某新闻被 fact-check 标 fake，怎么 retroactively 下架？"

A：

  • Fact-check DB 推 webhook to news service
  • Article 标 flagged=true (soft delete)
  • All user feeds revalidate (cache invalidate that article_id)
  • 如果已 push notification: 不撤回，但 article 详情显示 "fact-check 标 disputed"

Q7: "Multi-language clustering 怎么 work？"

A：用 multilingual embedding (LaBSE / mBERT)：同一概念跨语言 embed 接近。Cluster 不分语言，但 user 看 feed 时按 user lang filter cluster's articles。

9. 易错点 & 加分项

❌ 易错点

  1. 没有 dedup pipeline → feed 满是同样内容 5 遍
  2. All-in-one DB → 不知道哪用 ES / FAISS / Redis
  3. Crawler 不 polite → 被 source 封禁
  4. Personalization 现算 → p99 慢
  5. Multi-language 当 monolith → 中文用户看到英文
  6. 没说 breaking news pipeline → 重大新闻 1 小时才看到

✅ 加分项

  1. SimHash + Bloom filter + embedding cluster 三层 dedup
  2. Two-tower model 给 personalization
  3. HDBSCAN online clustering for events
  4. LaBSE multilingual embedding for cross-lingual cluster
  5. 3-tier cache (CDN + Redis + per-user)
  6. Source trust score 长期质量
  7. FAISS ANN for retrieval
  8. Newsworthiness ML classifier for breaking news
核心要点

STAFF vs SENIOR：能讲清"4 个 pipeline 串起来 + ML system + multi-language + multi-region"是 STAFF；只讲 crawler + DB 是 SENIOR。

10. Cheat Sheet

4 个 pipeline:
  1. Crawl: URL frontier → Worker → HTML parser → S3 raw
  2. Process: extraction → dedup (URL/SimHash/cluster) → embedding
  3. Personalize: two-tower DNN → FAISS retrieve → DNN rerank
  4. Deliver: feed cache → CDN → push for breaking

Dedup 3 layers:
  Layer 1: URL canonical + Bloom (exact)
  Layer 2: SimHash hamming < 3 (near)
  Layer 3: Embedding cluster cosine > 0.85 (semantic)

Storage:
  Raw: S3
  Hot 24h: Redis
  Search: Elasticsearch
  ANN: FAISS / Milvus
  User feed: precomputed, 1h cache

ML:
  - Two-tower user + item embed
  - HDBSCAN clustering
  - Newsworthiness classifier
  - Source trust score
  - LaBSE for multi-lang

Real-time:
  Breaking news: 5 min end-to-end
  Crawl high-priority every 30s
  Fan-out via Kafka

数字:
  1M articles/day → 100k unique events
  1B users, 500M DAU
  60k feed QPS sustained, 200k peak
  100 languages
